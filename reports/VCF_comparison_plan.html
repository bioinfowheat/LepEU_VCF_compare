Comparative assessment of a VCF set — design overview

Dataset: Pnapi_chr10_FR997704 — six VCFs across two mappers (BWA, NGM) × three downstream settings (cohort, hwe, nohwe), all called with bcftools.

Framing. Because you have the same reads and the same caller across all six VCFs, differences between files come from (a) mapping ambiguity / read placement, and (b) post-call filtering choices. That structure matters: the comparison should not just ask “do the VCFs agree?” but “where do they disagree (genomic context, allele frequency, coverage), and does that disagreement propagate into downstream population-genomic estimates?”. A site-level Venn diagram is the start, not the end.

2. Why these tiers, in popgen terms

TierQuestion it answersWhy it matters for popgen
0. NormalisationAre records comparable?bcftools/GATK represent indels and MNPs differently; without left-alignment + decomposition, “intersections” are artefactually low.
1. Per-file QCIs any one VCF anomalous on its own?Ts/Tv, het/hom, and the unfolded SFS catch systemic bias before you start comparing.
2. ConcordanceWhich sites & genotypes agree?Site agreement is permissive; genotype concordance is the popgen-relevant quantity because π, Fst, and PCA all depend on genotype calls, not just on “a SNP was called here”.
3. Genomic contextWhere do disagreements live?If discordance is concentrated in repeats, low-complexity, or near indels, it is a mapper artefact; if it is in HWE-deviating sites, it is a filter artefact.
4. Popgen estimatorsDo summaries differ?π, dxy, Fst, Tajima's D are what end up in the paper. Calculate with pixy using all-sites VCFs so missing data is handled correctly.
5. RobustnessDo conclusions change?The most defensible result is one where the top-ranked outlier windows / PCA structure agree across mappers and filtering regimes.
HWE filtering caveat. Hard HWE filters remove true variants under selection, in structured populations, or with sex-linked inheritance. The hwe vs nohwe comparison is therefore not just a technical filter test — it is a biological-assumption test. Expect Fst peaks to shrink under HWE filtering.

3. What VCF normalisation actually does

VCF is a text format with multiple legal ways to write the same biological variant. Two callers can be 100% biologically concordant and still share 0 lines after a naive grep or isec. Normalisation forces every record into a single canonical representation so that comparison is meaningful. bcftools norm does four things, and you should know what each is doing to your data:

3.1 Left-alignment of indels

In a repetitive context, an indel can be written at several positions and they are all biologically identical. Convention: shift it as far left as the reference allows.

ReferenceNaïve VCF (caller A)Naïve VCF (caller B)Left-aligned (canonical)
AAAAACGPOS=3 REF=A ALT=AA
(insert at pos 3)		POS=5 REF=A ALT=AA
(insert at pos 5)		POS=1 REF=A ALT=AA

Both callers inserted one A into a run of As — there is no biological way to tell which A. Without left-alignment, bcftools isec sees them as two different sites. After bcftools norm -f REF, they collapse to the same record.

3.2 Parsimony / trimming

The REF and ALT should share no common suffix and (after position adjustment) no common prefix beyond the anchor base. Trim them down to the minimal representation.

BloatedParsimonious
REF=CACAC ALT=CACREF=CAC ALT=C (suffix AC trimmed)
REF=GTTC ALT=GTACPOS+=2 REF=T ALT=A (it's a SNP, not an MNP)

3.3 Splitting multi-allelic sites (-m -any)

One site can be written as a single record with multiple ALTs, or as multiple records with one ALT each. Comparing across files only works if everyone uses the same convention. bcftools norm -m -any means “split everything down to bi-allelic”, which is what almost every downstream tool (pixy, plink, vcftools, snpEff) actually wants.

BEFORE (multi-allelic, one line):
chr1  100  .  A  G,T  .  PASS  .  GT  0/1  1/2

AFTER bcftools norm -m -any (two bi-allelic lines):
chr1  100  .  A  G    .  PASS  .  GT  0/1  1/.
chr1  100  .  A  T    .  PASS  .  GT  ./.  ./1

Notice that genotypes are re-encoded per ALT — that is the whole point. After this, every record represents exactly one alternative allele, so set operations and per-allele statistics are well-defined.

3.4 Decomposing MNPs / complex variants

An MNP like REF=AC ALT=GT at one position is actually two SNPs in a row. Different callers split or don't split these. bcftools norm -a (or piping through vt decompose_blocksub) breaks them into individual SNPs so that, again, the unit of comparison is identical across files.

3.5 The single command that does all of it

bcftools norm \
  -f reference.fa \      # left-align needs the reference sequence
  -m -any \              # split multi-allelics into bi-allelic records
  -c w \                 # warn (don't crash) if REF doesn't match the fasta
  --atom-overlaps . \    # break overlapping deletions cleanly
  -Oz -o out.norm.vcf.gz \
  in.vcf.gz
bcftools index -t out.norm.vcf.gz
FlagWhat it doesWhy it matters here
-f reference.faProvides the reference for left-alignment and REF checking.BWA and NGM can produce VCFs where the REF column was emitted with subtle differences (case, ambiguity codes). The fasta is the source of truth.
-m -anyDecomposes multi-allelics. +any would re-join them.Required for pixy, PLINK, and meaningful isec.
-c wSanity-check REF against fasta; warn on mismatch.Catches the easy disaster of running with the wrong reference build.
--atom-overlaps .Handles deletions that overlap other variants.Mapper differences around indels often produce overlapping records; this resolves them deterministically.
-OzOutput bgzipped VCF.Required for tabix-indexing.
Empirical rule of thumb. On the same dataset, naive site overlap between two callers is typically 70–85%. After normalisation, it jumps to 90–97%. That gap is almost entirely representation, not biology — and you do not want to confuse the two when interpreting your six VCFs.

3.0 Setup: variables & normalisation

VCFDIR=Pnapi_chr10_FR997704
REF=Pnapi_FR997704.fa                # supply: reference fasta, indexed (.fai)
CHR=FR997704.1
OUT=compare_out
mkdir -p $OUT/{norm,stats,isec,annot,pixy,plink,reports}

VCFS=(
  Pnapi_bwa_bcftools_cohort_FR997704.1.vcf.gz
  Pnapi_bwa_bcftools_hwe_FR997704.1.vcf.gz
  Pnapi_bwa_bcftools_nohwe_FR997704.1.vcf.gz
  Pnapi_ngm_bcftools_cohort_FR997704.1.vcf.gz
  Pnapi_ngm_bcftools_hwe_FR997704.1.vcf.gz
  Pnapi_ngm_bcftools_nohwe_FR997704.1.vcf.gz
)

# 0. Normalise EVERY file the same way before any comparison
for v in "${VCFS[@]}"; do
  base=$(basename $v .vcf.gz)
  bcftools norm -f $REF -m -any -Oz -o $OUT/norm/${base}.norm.vcf.gz $VCFDIR/$v
  bcftools index -t $OUT/norm/${base}.norm.vcf.gz
done

3.1 Tier 1 — per-file QC

# Stats per file (ts/tv, indel ratio, depth, quality, SFS)
for v in $OUT/norm/*.norm.vcf.gz; do
  bcftools stats -s - $v > $OUT/stats/$(basename $v .vcf.gz).stats
done

# Pretty multi-file plot (uses Python+matplotlib; ships with bcftools)
plot-vcfstats -p $OUT/stats/plots/ $OUT/stats/*.stats

# Per-sample missingness, heterozygosity, mean depth (vcftools)
for v in $OUT/norm/*.norm.vcf.gz; do
  out=$OUT/stats/$(basename $v .norm.vcf.gz)
  vcftools --gzvcf $v --missing-indv  --out $out
  vcftools --gzvcf $v --het           --out $out
  vcftools --gzvcf $v --depth         --out $out
done

Go beyond site overlap. Two VCFs can call the same site but disagree on the genotypes. Use hap.py or RTG vcfeval to get a confusion matrix per pair, stratified by region/MAF. This is what changes π. 
# Pairwise genotype concordance (one VCF as the "truth" reference)
TRUTH=$OUT/norm/Pnapi_bwa_bcftools_nohwe_FR997704.1.norm.vcf.gz
for q in $OUT/norm/*.norm.vcf.gz; do

  hap.py $TRUTH $q -r $REF -o $OUT/isec/happy_${name} \
         --no-roc --threads 8

3.4 Tier 4 — population-genomic estimators

For π, dxy, Fst use pixy, not vcftools --window-pi. pixy requires an all-sites VCF (variant + invariant) so that missing data and monomorphic positions are handled correctly. If you only have variant-only VCFs, regenerate with bcftools mpileup ... | bcftools call -m -f GQ --output-tags ... emitting all sites, or use GATK GenotypeGVCFs --include-non-variant-sites. 
# popmap.tsv:  sample_id  population
# All-sites VCFs (one per method), gzipped + tabix-indexed
for v in $OUT/norm/*.allsites.vcf.gz; do
  name=$(basename $v .allsites.vcf.gz)
  pixy --stats pi fst dxy \
       --vcf $v \
       --populations popmap.tsv \
       --window_size 50000 \
       --n_cores 8 \
       --output_folder $OUT/pixy/$name \
       --output_prefix $name
done

# vcftools complements: Tajima's D, LD decay
for v in $OUT/norm/*.norm.vcf.gz; do
  out=$OUT/pixy/$(basename $v .norm.vcf.gz)
  vcftools --gzvcf $v --TajimaD 50000 --out $out
  vcftools --gzvcf $v --geno-r2 --ld-window-bp 50000 --out $out
done

# PLINK: PCA, IBS, ROH — is sample structure stable across methods?
for v in $OUT/norm/*.norm.vcf.gz; do
  name=$(basename $v .norm.vcf.gz)
  plink2 --vcf $v --double-id --allow-extra-chr \
         --maf 0.05 --indep-pairwise 50 5 0.2 \
         --pca 10 --make-bed --out $OUT/plink/$name
done

3.5 Tier 5 — robustness of biological conclusions

This is the punchline. For each popgen metric, compute the cross-method agreement; this is what should appear in the paper.

# R: are the same 50 kb windows in the top 1% of Fst across methods?
library(data.table)
files <- list.files("compare_out/pixy", pattern="_fst.txt$",
                    recursive=TRUE, full.names=TRUE)
fst <- rbindlist(lapply(files, function(f){
  d <- fread(f); d$method <- sub("_fst.txt","",basename(f)); d
}))
top <- fst[, .(window=paste(chromosome,window_pos_1,sep=":"),
               method, avg_wc_fst)]
wide <- dcast(top, window ~ method, value.var="avg_wc_fst")

# (a) Spearman rank correlation of Fst across methods
cor(wide[,-1], use="pairwise.complete.obs", method="spearman")

# (b) Jaccard of top-1% outlier windows across methods
q99 <- sapply(wide[,-1], quantile, 0.99, na.rm=TRUE)
outliers <- mapply(function(col,thr) wide$window[which(col>=thr)],
                   wide[,-1], q99, SIMPLIFY=FALSE)
jacc <- outer(seq_along(outliers), seq_along(outliers),
              Vectorize(function(i,j){
                a<-outliers[[i]]; b<-outliers[[j]]
                length(intersect(a,b))/length(union(a,b))
              }))
dimnames(jacc) <- list(names(outliers), names(outliers))
jacc

4. The minimum viable comparison (if you only have a day)

  1. Normalise all six VCFs (bcftools norm -m -any -f REF).
  2. bcftools stats + plot-vcfstats — one PDF showing ts/tv, SFS, het/hom per file.
  3. bcftools isec → UpSet plot of site overlap, coloured by mapper vs filter axis.
  4. One SnpEff run on the union set; cross-tabulate impact category × method-of-origin.
  5. pixy π in 50 kb windows per method → Spearman correlation matrix across methods. Report this matrix.

If methods agree on (3) and the correlation matrix in (5) is >0.9 off-diagonal, your biology is robust to method choice. If not, the cells where they disagree are the story.

6. Things easy to get wrong

  • Comparing un-normalised VCFs. Multi-allelic vs split, left vs right alignment, MNPs — all inflate apparent disagreement. Always bcftools norm -m -any first.
  • Using variant-only VCFs for π / dxy. Underestimates monomorphic denominator and inflates diversity — use all-sites VCFs with pixy.
  • Treating HWE-filtering as cosmetic. It removes biology (selection, structure). Always report popgen estimates with and without it.
  • Equating site recall with genotype recall. A SNP can be called in both VCFs with different GTs — invisible to isec, lethal to π.
  • Ignoring callable mask. Restrict all comparisons to the intersection of per-method callable regions (mosdepth-based mask is fine); otherwise mapper-coverage differences masquerade as variant differences.
