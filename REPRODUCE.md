# REPRODUCE.md — full guide to rebuilding this analysis & report

This document explains **how the whole thing works end to end**, so you can:

- **(A)** generate a similar HTML report for a **new set of VCFs**, or
- **(B)** **rebuild the exact demo** from scratch on a **different computer**, or
- **(C)** hand the whole task to a **fresh Claude Code instance** and have it recreate
  everything (a ready-to-paste briefing is in §8, and the gotchas it must know are in §9).

It complements the per-step folder READMEs (`00_…`–`07_…`), which document each method
in depth. Read this first for the big picture and the exact run order.

---

## 1. What this project does

Compare a set of VCFs **made from the same sequencing reads by different methods**, at two
levels — (i) the SNPs called, and (ii) the downstream population-genomic summaries — and
assemble everything into **one self-contained HTML report**
(`reports/VCF_comparison_results_v3.html`, all figures embedded as base64).

Test case (the demo): *Pieris napi*, chromosome **FR997704.1** (= RefSeq `NC_062243.1`,
chr 10), 14 samples in two populations of 7 (**A** vs **P**), six VCFs =
`{bwa, ngm} mapper × {cohort, hwe, nohwe} filter`, all called with bcftools.

The report sections: 0 intro · 1 per-file QC · 2 site-set comparison (isec/UpSet/Jaccard)
· 3 annotation (bcftools csq) · 4 sliding-window popgen (π, Fst-WC, Fst-Hudson, dxy,
diagnostics, combined scan, depth) · 5 normalisation effects · 6 integrative (enrichment,
Ts/Tv) · 7 bottom line.

## 2. Inputs you must supply (NOT in this repo)

The repo carries **code + figures + the HTML report only**. The heavy inputs are
git-ignored (see `.gitignore`) and must be provided:

| input | needed for | notes |
|-------|-----------|-------|
| the VCFs (bgzipped + `.tbi`) | everything | ideally **all-sites** (variant + invariant) so pixy works; must carry `FORMAT/DP` for the depth step |
| reference FASTA (`.fa` + `.fai`) | norm, csq, pixy, depth | **contig names must match the VCFs' `#CHROM`** (see §5 gotcha) |
| GFF3 (sorted, bgzipped, tabix) | annotation (csq) | same coordinate system as the FASTA |
| `popmap.tsv` | pixy (π/Fst/dxy) | two columns: `sample<TAB>population`; sample IDs must match `bcftools query -l` |

Demo data layout (on the original machine):
```
VCF_compare/                      # working dir (NOT a git repo)
├── Pnapi_chr10_FR997704/         # the 6 VCFs (+ .tbi)   ~560 MB each
├── genome/                       # full genome + extracted single-chromosome ref + GFF
├── compare_out/                  # ALL working outputs (git-ignored) — regenerated by the scripts
├── scripts/                      # the working copies of the scripts
└── LepEU_VCF_compare/            # THIS repo (the portable, shareable artifact)
```

## 3. Environment setup from scratch (new computer)

Two toolchains: a conda/mamba env for the CLI tools + Python, and system R for UpSetR.

```bash
# 3a. conda/mamba env (bcftools, vcftools, pixy, bedtools, samtools, htslib, pandoc, python stack)
bash environment/create_env.sh
mamba activate vcfcompare
ENV_BIN=$(dirname $(which bcftools))     # remember this path; scripts reference it

# 3b. R + UpSetR (CRAN) — used by the two UpSet .Rmd files in step 03
mkdir -p ~/R_libs_vcfcompare
Rscript -e '.libPaths("~/R_libs_vcfcompare"); install.packages(c("UpSetR","data.table","rmarkdown"), repos="https://cloud.r-project.org")'
# rmarkdown needs pandoc; we installed it in the conda env above and point R at it with:
#   Sys.setenv(RSTUDIO_PANDOC="<conda-env>/bin")   (the render commands do this)
```

**macOS / Apple-Silicon gotcha:** the bioconda `ensembl-vep` build aborts with
`dyld: missing symbol called`. We use **`bcftools csq`** for annotation instead (same
GFF + FASTA, same consequence taxonomy). On Linux / via VEP-Docker, VEP proper works —
swap it into step 04. (`plink2`, used for optional PCA, also isn't in the channel set;
install separately only if you want it — the report doesn't use it.)

> **Paths to fix on a new machine.** The scripts in the numbered folders and `lib/` use
> absolute paths from the original machine (`/Users/stockholmbutterflylab/...`,
> `~/R_libs_vcfcompare`, the conda env bin). Edit the variables at the **top of each
> script** (and the `.libPaths`/`RSTUDIO_PANDOC` in the `.Rmd`s) to your machine. The
> `run_on_your_data/config.sh` route centralises all of this into one file — prefer it
> for new datasets (§7).

## 4. Two ways to run

- **Mode A — your own VCFs (recommended for new data):** `run_on_your_data/` — edit one
  config file, run one script. Does normalise → stats → N-way isec → UpSet + Jaccard →
  csq → pixy for *any* number of VCFs. See §7 and `run_on_your_data/README.md`.
- **Mode B — reproduce the full demo report:** run the ordered pipeline in §6, then
  `lib/make_report.py`. This regenerates every figure and the HTML exactly.

---

## 5. The contig-name gotcha (do this first)

The demo VCFs label the chromosome `FR997704.1` (ENA accession) but the downloaded
genome uses RefSeq `NC_062243.1`. `bcftools norm`, `bcftools csq` and `pixy` all need the
reference contig name to **equal** the VCF `#CHROM`. `00_reference_prep/prepare_reference.sh`
extracts the one chromosome and **renames its FASTA header** to the VCF name, and does the
same for the GFF (then sorts+bgzips+tabix-indexes it). If your VCFs and reference already
agree on contig names, skip this. To find your chromosome's RefSeq id:
`grep 'chromosome=10;' genome.gff | head -1`.

## 6. Full pipeline, in dependency order (Mode B)

All commands assume `mamba activate vcfcompare` and that you've edited the path variables.
Outputs accumulate under `compare_out/`. `make_report.py` (last step) reads everything
from `compare_out/plots/`.

```bash
# 0. single-chromosome reference + GFF whose names match the VCFs
bash 00_reference_prep/prepare_reference.sh

# 1. normalise every VCF (bcftools norm -m -any: left-align, trim, split multi-allelics)
bash 01_normalization/normalize.sh            # -> compare_out/norm/*.norm.vcf.gz

# 2. per-file QC (bcftools stats), raw + normalised
bash 02_per_file_qc/per_file_stats.sh         # -> compare_out/stats_{raw,norm}/

# 3. site-set comparison
bash 03_site_comparison/run_isec.sh           # -> compare_out/isec_{raw,norm,all12}/sites.txt
python 03_site_comparison/jaccard.py          # -> figures jaccard_{raw,norm,all12}
#    UpSet plots (R): point .libPaths + RSTUDIO_PANDOC inside the render call
Rscript -e '.libPaths("~/R_libs_vcfcompare"); Sys.setenv(RSTUDIO_PANDOC="'"$ENV_BIN"'"); rmarkdown::render("03_site_comparison/upset_tier2.Rmd", knit_root_dir=getwd())'
Rscript -e '.libPaths("~/R_libs_vcfcompare"); Sys.setenv(RSTUDIO_PANDOC="'"$ENV_BIN"'"); rmarkdown::render("03_site_comparison/upset_all12.Rmd", knit_root_dir=getwd())'

# 4. annotation (bcftools csq using the extracted GFF) -> compare_out/vep/*.csq.tsv
bash 04_annotation/annotate_csq.sh

# 5. population genomics (pixy), all 6 VCFs x {raw,norm}
bash 05_popgen/pixy.sh                         # π / Fst(W&C) / dxy  -> compare_out/pixy_{raw,norm}/
bash 05_popgen/pixy_hudson.sh                  # Fst Hudson           -> compare_out/pixy_hudson_{raw,norm}/

# 5b. read depth
bash 05_popgen/depth_windows.sh                # vcftools --site-mean-depth -> compare_out/depth/*.ldepth.mean
python 05_popgen/depth_windows.py              # bins to windows -> depth_per_{file,method}, depth_per_window.tsv
python 05_popgen/depth_per_individual.py       # per-sample DP traces (bwa_hwe vs bwa_nohwe panels)

# 6. plotting engine + diagnostics (these populate most of compare_out/plots/)
python lib/analyze_and_plot.py                 # tiers 1–5 plots; combined scan (depth+π+Fst-WC+Fst-Hudson+dxy)
python lib/v3_additions.py                     # 12×12 Jaccard; per-method raw-vs-norm π
python 07_integrative/integrative_analysis.py  # consequence enrichment; Ts/Tv; one-vs-five π
python 05_popgen/fst_hudson_compare.py         # WC-vs-Hudson scatter; Hudson tracks + correlation

# 7. assemble the HTML report (embeds every PNG/TSV in compare_out/plots/)
python lib/make_report.py                      # -> reports/VCF_comparison_results_v3.html
```

Notes:
- Heavy steps (norm, pixy) are parallelised inside the scripts; on the original 28-core
  M3 Ultra the whole run is well under an hour.
- pixy is the long pole. The depth `.ldepth.mean` files are ~460 MB each — fine on disk,
  git-ignored.

## 7. Adapting to a new VCF set (Mode A — generate a similar HTML)

For the **core comparison** (normalise, intersect, UpSet, Jaccard, annotate, pixy) on any
VCFs, use `run_on_your_data/`:

```bash
cd run_on_your_data
#   edit config.sh: VCFS=(...), LABELS=(...), REF, GFF, POPMAP, OUTDIR, ENV_BIN, R_LIBS_DIR
bash run_all.sh
```
`LABELS` (same length/order as `VCFS`) defines the bit/column order in every plot. The
UpSet (`upset_generic.Rmd`) and Jaccard (`jaccard_generic.py`) read `labels.txt` and work
for any N call sets. See `run_on_your_data/README.md` for the deep walkthrough of the
intersection bitmask and the RMD.

To regenerate the **full multi-section HTML** for new data, the lower-friction path is to
run Mode A, then run the `lib/` + `05_popgen/` + `07_integrative/` plotting scripts and
`make_report.py` against your `OUTDIR`. **Dataset-specific things to change** in those
scripts if you adapt them by hand:

| what | where | change to |
|------|-------|-----------|
| the 6 method names / filename globbing | `lib/*.py`, `05_popgen/*.sh`, `07_integrative/*.py` | your labels |
| contig name `FR997704.1` / length | `00_reference_prep`, plot x-axis labels | your chromosome |
| populations `A` / `P` | `popmap.tsv`, pixy parsing | your populations |
| the `*_hwe` dashed-line rule | `method_ls()` in `lib/analyze_and_plot.py` & `depth_*` | your "middle" category, or remove |
| window size 50000 | pixy + all binning scripts | your window |

## 8. Rebuilding with a fresh Claude Code instance

On a new computer: `git clone` this repo, put your VCFs + reference + GFF + popmap beside
it, then start Claude Code in the parent folder and paste a briefing like:

> *I have the repo `LepEU_VCF_compare` and a set of VCFs in `<dir>` made from the same reads
> by different methods, plus a reference FASTA, a GFF, and a popmap. Read
> `LepEU_VCF_compare/REPRODUCE.md` and the per-folder READMEs. Reproduce the comparative
> analysis and assemble the single HTML report (`make_report.py`). Work in a conda env
> per `environment/create_env.sh`. First fix any contig-name mismatch between the VCFs and
> the reference (§5). Use all-sites VCFs for pixy and do NOT pre-split multi-allelics for
> pixy. Use `bcftools csq` (not VEP) for annotation, and keep intergenic variants. Run
> tiers in the order in §6, then build the report. Tell me the headline findings.*

Then let it work tier by tier. The non-obvious knowledge it must preserve is in §9 — point
it there explicitly.

## 9. Gotchas & lessons learned (read before reproducing)

These were discovered the hard way; preserving them avoids silently wrong results.

1. **Contig names (§5).** RefSeq vs ENA mismatch silently breaks norm/csq/pixy. Rename the
   reference FASTA/GFF header to match the VCF `#CHROM`.
2. **pixy needs all-sites VCFs** (variant + invariant) for a correct π/dxy denominator.
3. **Do NOT pre-split multi-allelics (`norm -m -any`) before pixy.** Splitting inflates π
   and dxy by ~12% (pixy counts each split record's differences separately). Use the
   normalised/split VCFs for isec + annotation, but the **un-split (raw) all-sites VCFs**
   for π/dxy/Fst. (Fst is a ratio, so it largely cancels this.)
4. **Annotation: `bcftools csq | bcftools query`, not `+split-vep`.** `+split-vep` silently
   *drops* variants with no consequence tag — i.e. all intergenic SNPs — making it look
   like ~98% of SNPs are intronic. `bcftools query` keeps them as `.` (= intergenic).
   `intronic` ≠ `intergenic` — they are separate classes.
5. **Normalisation does NOT shrink the SNP union.** It slightly *increases* the union
   (splitting multi-allelics → more bi-allelic records); the meaningful metric is the
   **fraction of sites shared by all methods**, which rises (68.8% → 71.9% in the demo).
   Read the shared-fraction column, not the union size.
6. **Depth from `vcftools --site-mean-depth` on the raw all-sites VCFs** (depth at every
   base, not just SNPs). Depth is mapper-driven; in the demo the three filters within a
   mapper are byte-identical (md5-verified) — but compute per file, don't assume.
7. **`goleft` can't run here.** Its coverage tools (indexcov/covstats) need BAM/CRAM (or
   `.bai`/`.crai`); we only have VCFs. The depth step uses VCF `FORMAT/DP` instead. If you
   later get the BAMs, `goleft indexcov` is a great caller-independent complement.
8. **The ≈7 Mb depth spike** (to ~22×, driven by a subset of individuals) sits under the
   Fst peak / π-dxy crash → a collapsed-repeat artefact, not selection. Overlay depth on
   any scan before trusting an Fst outlier; consider masking windows > ~2× median depth.
9. **Fst estimator matters.** WC (default) vs Hudson (`--fst_type hudson`) differ most at
   the outlier windows; Hudson is somewhat more method-robust. Report which you used.
10. **Plot convention:** the middle shade of each colour gradient (`*_hwe`) is drawn dashed
    in the sliding-window tracks so the three lines per mapper are distinguishable
    (`method_ls()`).
11. **git identity / commits.** Commits were made with per-command identity overrides
    (`git -c user.name=... -c user.email=...`) to avoid touching global config. The
    `.gitignore` keeps all large data (VCF/BAM/FASTA/GFF and the whole `compare_out/`) out
    of the repo — re-check it before committing on a new machine.
